Cyber Security Reference

OWASP risk frameworks, the Essential Eight, and a full glossary — all in plain English for everyday Australian organisations.

The OWASP Top 10 for Web Applications (2021) — the ten most critical security risks facing web applications, ordered by how commonly they occur and how much damage they cause. If you build or buy web software, these are the gaps attackers look for first.

owasp.org/www-project-top-ten ↗
  1. A01 Broken Access Control

    When a system doesn't properly check whether a user has permission to see or do something. An employee accesses another person's files. A regular user reaches admin functions. A customer changes an ID in a URL to view someone else's order. The #1 risk — found in over 94% of tested applications.

  2. A02 Cryptographic Failures

    Sensitive data — passwords, credit cards, health records — transmitted or stored without proper encryption, or protected using outdated algorithms an attacker can crack. The problem isn't always "no encryption" — it's often encryption done badly. Previously called "Sensitive Data Exposure."

  3. A03 Injection

    Attackers trick an application into running malicious code by slipping it into user input — a search box, a login form, a URL parameter. SQL injection (accessing or destroying databases), command injection (running system commands), and cross-site scripting (XSS) all fall here. Classic, common, and still devastating.

  4. A04 Insecure Design

    Security wasn't part of the design — it was bolted on later, or not at all. No threat modelling, no secure design patterns, no security requirements set before development began. You can patch implementation bugs, but you can't patch a fundamentally broken architecture. New in 2021 — acknowledges that the root cause is often design, not code.

  5. A05 Security Misconfiguration

    Default passwords left in. Unnecessary features turned on. Verbose error messages exposing system details. Cloud storage set to public. Permissions too broad. This is the most common gap — not a clever exploit, just things not configured correctly. 90% of applications had some form of misconfiguration in testing.

  6. A06 Vulnerable and Outdated Components

    Using libraries, frameworks, plugins, or operating systems with known vulnerabilities. Modern applications are mostly composed of third-party code — if that code has a known CVE and you haven't updated, attackers know exactly where to look. The Log4Shell vulnerability in 2021 is a defining example of this category.

  7. A07 Identification and Authentication Failures

    Weak login systems: no multi-factor authentication, common passwords allowed, no lockout after failed attempts, session tokens that don't expire, passwords stored in plain text. When authentication can be bypassed or brute-forced, everything behind the login screen is exposed.

  8. A08 Software and Data Integrity Failures

    Trusting code or data from untrusted sources without verification. Includes supply chain attacks — a malicious package slipped into your build pipeline, auto-updates delivered without integrity checks, CI/CD pipelines without security controls. The SolarWinds attack is the defining real-world example.

  9. A09 Security Logging and Monitoring Failures

    Insufficient logging, logs that aren't monitored, no alerts when attacks happen, logs that can be tampered with. The average time to detect a breach is over 200 days. You can't respond to an incident you don't know about — and without logs, you can't even prove what happened after the fact.

  10. A10 Server-Side Request Forgery (SSRF)

    Tricking a server into making requests to internal systems or services the attacker shouldn't be able to reach directly. In cloud environments, this can expose instance metadata services containing credentials. Increasingly dangerous as more infrastructure moves to cloud architectures. New in 2021, added based on industry survey data before exploitation rates caught up.

The OWASP API Security Top 10 (2023) — the ten most critical risks specific to APIs (Application Programming Interfaces). APIs are the backbone of modern software — mobile apps, web apps, and third-party integrations all rely on them. They're also a major and often underprotected attack surface.

owasp.org/API-Security ↗
  1. API1 Broken Object Level Authorization

    The API doesn't check whether you have permission to access a specific object — just that you're logged in. Change a customer ID or order number in a request and you see someone else's data. The most common and impactful API vulnerability. Every object-level access needs its own authorization check.

  2. API2 Broken Authentication

    Weak or missing authentication on API endpoints. No tokens required, tokens that never expire, tokens that can't be revoked, no protection against credential stuffing. APIs exposed to the internet without authentication are essentially public databases.

  3. API3 Broken Object Property Level Authorization

    The API returns more fields than needed (exposing sensitive data the user shouldn't see) or allows updates to fields that shouldn't be changeable. A user profile endpoint that returns salary or admin flags. A form that accepts fields not shown in the UI but processed by the server.

  4. API4 Unrestricted Resource Consumption

    No rate limiting or resource controls. Attackers hammer your API, run up cloud costs, extract your entire dataset one request at a time, or trigger expensive backend operations at scale. Also includes missing limits on file size, record count, or request frequency.

  5. API5 Broken Function Level Authorization

    Admin-only API functions are accessible to regular users because authorization is only enforced on the frontend, not the API itself. A regular user calls an admin endpoint directly and gets access. Different from object-level authorization — this is about which functions you can call, not which data you can see.

  6. API6 Unrestricted Access to Sensitive Business Flows

    Automating business actions that should be limited by design — bulk account creation for spam, scalping limited-release tickets, loyalty point abuse, mass voucher redemption. The API works as designed — the problem is there are no limits preventing automated abuse at scale.

  7. API7 Server Side Request Forgery

    Tricking the API into making requests to internal systems on behalf of the attacker — cloud metadata endpoints, internal admin interfaces, databases that aren't meant to be internet-facing. As for the web equivalent, increasingly dangerous in cloud environments.

  8. API8 Security Misconfiguration

    Default configurations, unnecessary HTTP methods left enabled (DELETE, PUT on read-only endpoints), verbose error messages that reveal stack traces, missing transport security (TLS), CORS configured too broadly, missing security headers. Same category as the web list — just as common, just as preventable.

  9. API9 Improper Inventory Management

    Old API versions left running alongside the new ones. Undocumented internal endpoints. Test environments exposed to the internet. Third-party APIs included in production without vetting. You can't secure what you don't know exists. API sprawl is real — organisations often don't know how many API endpoints they have.

  10. API10 Unsafe Consumption of APIs

    Blindly trusting data from third-party APIs without validation or sanitisation. If a partner API is compromised, malicious data flows straight into your application. Redirect attacks, injection via API responses, unvalidated data used in queries. Your application is only as safe as the APIs it trusts.

The OWASP Top 10 for Large Language Model Applications (2025) — the ten most critical risks when building or using AI applications. As AI tools move from experimental to operational across Australian organisations, these are the risks that security teams and IT leaders need to understand.

owasp.org LLM Project ↗
  1. LLM01 Prompt Injection

    Crafting inputs that override an AI's instructions — making it ignore its guidelines, reveal its system prompt, or take actions its developers didn't intend. "Ignore previous instructions and…" is the classic form, but injections hidden in documents, web pages, or email content are increasingly common and much harder to detect.

  2. LLM02 Sensitive Information Disclosure

    The AI reveals information it shouldn't — from its system prompt, its context window, or its training data. This includes confidential business logic embedded in prompts, personally identifiable information from training datasets, or proprietary data loaded into a RAG system. The right question, asked the right way, can extract information the developer assumed was private.

  3. LLM03 Supply Chain Vulnerabilities

    Risks in the models, datasets, plugins, and integrations you rely on to build AI applications. A poisoned training dataset, a compromised model weight, a malicious plugin in an AI marketplace — these affect every application built on top of them. You inherit the security posture of every component in your AI stack.

  4. LLM04 Data and Model Poisoning

    Corrupting training or fine-tuning data to change how a model behaves — introducing biases, backdoors, or incorrect outputs on specific inputs. Hard to detect and potentially catastrophic: a model that looks fine in testing but behaves badly in targeted scenarios. Fine-tuning on untrusted data is the most accessible vector.

  5. LLM05 Improper Output Handling

    Feeding AI output directly into downstream systems — code execution environments, databases, HTML renderers, email clients — without validation or sanitisation. If the AI generates malicious JavaScript that gets rendered in a browser, or SQL that gets executed without checking, you've handed the attacker a path through your AI into your systems.

  6. LLM06 Excessive Agency

    Giving an AI agent too much capability to act on its own — sending emails, deleting files, making purchases, calling external services — without appropriate oversight, approvals, or limits. The more autonomy an AI has, the more damage a prompt injection or jailbreak can cause. Least privilege applies to AI the same way it applies to users.

  7. LLM07 System Prompt Leakage

    Users extracting the hidden instructions given to an AI — the system prompt that defines its persona, restrictions, and context. If your system prompt contains sensitive business logic, confidential information, or details about your internal architecture, it may be extractable through carefully crafted questions. Don't treat the system prompt as a security boundary.

  8. LLM08 Vector and Embedding Weaknesses

    Vulnerabilities in RAG (retrieval-augmented generation) systems and vector databases. Poisoned documents can influence what the AI retrieves and how it responds. Access controls on the vector database may differ from what the AI assumes. If an attacker can influence what documents are indexed, they can influence the AI's answers.

  9. LLM09 Misinformation

    LLMs generate confident, fluent, plausible-sounding text — including when that text is completely wrong. Hallucinated citations, incorrect medical or legal advice, fabricated statistics. Serious risk when AI output is used for decisions without human verification. Not a bug — it's a fundamental characteristic of how these models work.

  10. LLM10 Unbounded Consumption

    No limits on tokens, API calls, or compute — enabling denial of service attacks, runaway cloud costs, or resource exhaustion. Especially risky for publicly accessible AI applications where users can trigger expensive operations without rate limiting. Also includes model theft through excessive querying — extracting model behaviour to replicate it.

The Essential Eight — the Australian Signals Directorate's baseline cyber security controls for Australian organisations. Not a compliance checkbox — a prioritised set of mitigations designed to prevent the most common attacks. ACSC recommends all Australian organisations implement these at Maturity Level 2 as a minimum.

cyber.gov.au Essential Eight ↗
1 Application Control

Only allow approved applications to run on your systems. Executables, scripts, libraries — if they're not on the approved list, they don't run. Stops malware and unauthorised software before it can execute. ACSC's single highest-priority mitigation — prevents the vast majority of malware incidents.

Prevents: malware execution, ransomware, unauthorised software
2 Patch Applications

Keep applications updated — particularly internet-facing software and anything that handles files or content (browsers, Office, PDF readers). Most attacks exploit known vulnerabilities with available patches. If a patch exists and you haven't applied it, attackers know exactly where to look.

Prevents: exploitation of known vulnerabilities in third-party software
3 Configure Microsoft Office Macro Settings

Restrict or disable macros in Office files — particularly macros from the internet. Malicious macros embedded in Word documents and Excel spreadsheets are one of the most common malware delivery methods. Users rarely need them. Blocking untrusted macros removes a major attack vector with minimal business impact.

Prevents: macro-based malware, ransomware delivered via email attachments
4 User Application Hardening

Block high-risk features in the applications your people use every day. Disable Flash (effectively dead, still occasionally exploited), Java in browsers, and ads from untrusted sources. Configure browsers and email clients to block scripts and content that users don't need. Reduces the attack surface of your most-used software.

Prevents: drive-by downloads, malvertising, browser-based exploits
5 Restrict Administrative Privileges

Give people only the access they need to do their job. No standing admin accounts for day-to-day work — use a separate privileged account when needed and log off after. Admin accounts are the keys to the kingdom. An attacker who compromises a standard user account gets far less than one who compromises an admin. Fewer admins, narrower blast radius.

Prevents: privilege escalation, lateral movement, damage from compromised accounts
6 Patch Operating Systems

Keep Windows, macOS, and Linux operating systems current. Operating system vulnerabilities can give attackers a path to full system or network control. Prioritise internet-facing systems and anything that's critical to operations. Unpatched operating systems — especially those past vendor support — are a persistent high-value target.

Prevents: OS-level exploitation, worm propagation, full system compromise
7 Multi-factor Authentication

Require more than a password to log in — a second factor like an authenticator app, hardware token, or passkey. MFA stops the vast majority of credential-based attacks, even when passwords have been stolen, guessed, or phished. Apply it everywhere: email, remote access, cloud services, and critical systems. Password alone is no longer sufficient.

Prevents: account compromise from stolen or phished credentials
8 Regular Backups

Regular, tested, offline backups of important data, software, and system configuration. The difference between a ransomware incident and a ransomware disaster is whether you can restore. Backups must be disconnected from the network (so ransomware can't encrypt them too) and regularly tested (so you know they work before you need them).

Prevents: unrecoverable data loss, ransomware leverage, extended downtime
